# Requirement Version 2 — Light workflow persistence

- Version: 2
- Previous version: `v1.md`
- Authorized by: 006006
- Authorized action: approve
- Provider: codex
- Status: approved for prompt generation and regulated execution

## Requirements carried forward

1. `light` workflow must not create or retain `.agrimap-agent/tasks/<task-id>/` or task artifacts such as `brief.md`, `checklist.md`, `qa.md`, and `result.md`.
2. `light` workflow must retain durable audit evidence only under `.agrimap-agent/logs/**` and `.agrimap-agent/memory/**`; ignored `runtime/**` session and active markers may remain operational state but are not durable evidence.
3. Every operation running at `depth=light` follows this persistence rule; history remains a direct light query that does not start task state.
4. `start`, `checkpoint`, `validate`, `complete`, `close`, and `history` must support light tasks without reading task artifacts.
5. Light validation must prove requester attribution, workflow/execution identity, created and terminal audit events, current/recent memory, claimed files, verification evidence, outcome, and reason from logs/memory.
6. `standard` and `regulated` workflow behavior and task artifacts remain unchanged.
7. Existing `.agrimap-agent/tasks/**` history must not be deleted or migrated destructively.
8. History must return light attribution and audit-storage semantics from logs/memory and explicitly mark task-artifact paths as not applicable.
9. Canonical runtime, artifact schema, lifecycle/operation contracts, hooks, tests, documentation, and generated provider mirrors must agree after sync.
10. Existing uncommitted SQL preflight files and untracked `AGENTS.md` are outside this requirement and must not be modified, removed, staged, or overwritten.
11. Verification must cover new light behavior, unchanged standard/regulated behavior, history attribution, close/cancel paths, package validation, sync drift, and the full existing test suite.
12. No commit, publish, release, global-cache mutation, destructive migration, or deletion is authorized by this version.

## Approved approach

- Depth-based persistence is canonical; no optional `--no-task-artifacts` flag.
- Light completion updates terminal evidence in logs/memory and archives current memory to recent without creating a result artifact.
- Specific missing or invalid standard/regulated task artifacts remain blocking.

## Changes from Version 1

- Added explicit all-light-operation coverage, history not-applicable semantics, terminal outcome/reason evidence, `AGENTS.md` protection, and approval of the depth-based approach.
