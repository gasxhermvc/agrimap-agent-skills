## Requirements carried forward from Version 3

1. Every `workflow_depth=light` operation creates no task directory and no task artifacts. It retains only the non-task operational evidence defined by this version.
2. Ignored `runtime/**` markers may remain operational state but are not durable evidence.
3. Light start/checkpoint/validate/complete/blocked/cancelled/history paths work without task artifacts and retain requester, executor, files, verification, outcome, reason, and concerns in logs/memory/report evidence.
4. Existing task, log, memory, prompt, and requirement history is never deleted or destructively rewritten; readers remain backward compatible.
5. Canonical runtime, schemas, lifecycle/operation contracts, hooks, tests, documentation, generators, and provider mirrors remain synchronized.
6. No commit, publish, release, global-cache mutation, or destructive history migration is authorized during implementation.
7. Golden artifacts remain teaching examples/templates and provenance evidence. Incoming user/project input is never required to match a golden file or hash.
8. Normalized pattern contracts, owner decisions, active schema/callers/tests, and deployed compatibility govern transformation. Applicable golden examples provide optional structural guidance only.
9. `raw-immutable` golden material remains analysis/provenance evidence and is not a released-runtime template by default.
10. Raw-byte hash, manifest coverage, duplicate-content, and package-integrity verification remains development/CI/package validation only.
11. Released runtime preflight must not import, call, or depend on `verifyGolden`, `HASH_MISMATCH`, or package-wide golden integrity state.
12. Runtime preflight validates task input identity and selects optional guidance. Missing/inapplicable examples warn or fall back to normalized rules rather than globally blocking work.
13. Hard runtime blocks remain limited to invalid required input, missing authoritative schema/caller evidence when necessary, or unresolved material behavior/data ambiguity.
14. Conflict precedence, SQL guidance, operation contracts, tests, docs, and generated mirrors must not say that golden examples override applicable active contracts or that package integrity blocks runtime inspection/generation.
15. `npm run verify:golden` remains strict and passing as a development/CI/package command.
16. The safe PowerShell cache-clear command must resolve and verify the exact Codex plugin cache target under `%USERPROFILE%\.codex\plugins\cache\agrimap-agent-skills` before recursive removal, and must be documented for use only after a fixed package is installed/reinstalled.

## Workflow-depth and task lifecycle decisions

17. Keep the canonical enum `workflow_depth=light|standard|regulated`; do not introduce `full` as a depth value.
18. `light` is artifactless with respect to `.agrimap-agent/tasks/**`. `standard` and `regulated` are the only task-producing depths.
19. Every activated operation receives a `run_id=ddHHmmss`. For `standard|regulated`, `task_id=run_id`; for `light`, `task_id=null`.
20. IDs and `YYYY-MM` path partitions use the configured project timezone, default `Asia/Bangkok`. Machine timestamps use ISO-8601 UTC with milliseconds.
21. ID allocation is atomic. An occupied ID returns `RUN_ID_COLLISION`; two runs must never share or merge state. Continuing work reuses the existing run/task ID and original start month.
22. Active task path is `.agrimap-agent/tasks/YYYY-MM/<task_id>/`. Completed tasks move to `.agrimap-agent/tasks/complete/YYYY-MM/<task_id>/`. Cancelled tasks move to `.agrimap-agent/tasks/cancelled/YYYY-MM/<task_id>/`.
23. A completed task folder contains `brief.md`, `analysis.md`, `checklists.md`, `qa.md`, and `result.md`. This is a completion set written by phase, not an empty start scaffold. `qa.md` may explicitly record not-applicable when the canonical QA contract permits it.
24. Blocked/partial tasks remain active and retain current memory. Cancelled tasks are terminal and archive under `tasks/cancelled`.
25. Task finalization and movement are idempotent and recoverable after interruption. History searches active, complete, cancelled, and legacy task locations.

## Memory lifecycle decisions

26. Canonical paths are `.agrimap-agent/memory/recent/YYYY-MM/<run_id>-<slug>.md`, `.agrimap-agent/memory/current/YYYY-MM/<run_id>-<slug>.md`, and `.agrimap-agent/memory/project.md`.
27. Every activated operation has exactly one concise recent journal. “Always write recent” means durable decisions, state, evidence pointers, and outcomes—not transcript, raw tool output, every prompt invocation, step-by-step activity, hidden chain-of-thought, or unsourced facts.
28. Current memory is the resumable state for active or blocked work. Milestones atomically update the same current file. Completed/cancelled work removes current; blocked/partial work retains it with reason and next action.
29. Every successful completion appends or updates one deduplicated, dated, short project note with the run/task/report pointer. Blocked/partial/cancelled work does not claim project completion.
30. Project-wide facts/owner decisions may be promoted earlier only with a source pointer and validity state. Recent retention remains configurable at 10–30 days, default 30; pruning never deletes current, project, tasks, logs, reports, requirements, or raw prompts.

## Logs, prompts, and reports

31. Canonical new log path is `.agrimap-agent/logs/YYYY-MM/YYYY-MM-DD.jsonl`, partitioned by project-local date and protected by a cross-process append lock.
32. New log keys use snake_case and include at minimum: `schema_version`, `timestamp`, `execution_id`, `task_id`, `workflow_depth`, `requester`, `requester_id`, `identity_source`, `model`, `model_label`, `role`, `agent`, `provider`, `event`, `log_type`, `message`, `files`, and `verification`.
33. Light log records use `task_id:null`. `event` is a closed lifecycle enum; `log_type` is one of `request|action|decision|verification|result|error`. Logs are concise machine events, not project documentation, prompt templates, memory injection, or human reports.
34. History reads both the new daily snake_case schema and existing per-task camelCase logs. It preserves attribution semantics, invalid-line reporting, storage status, and legacy artifact pointers without destructive conversion.
35. `.agrimap-agent/prompts/YYYY-MM/<conversation_id>/<context>.md` stores only raw owner/requester prompts supplied by the host, with ISO timestamp, a `Command:` marker, verbatim text, and deterministic separators. It never stores task folders or generated executor/QA prompts.
36. Generated execution instructions, when needed, live outside `prompts/`, under `.agrimap-agent/instructions/YYYY-MM/<task_id>/`.
37. The repository-local AGENTS.md rule that writes `agrimap-agent/prompts/history/<yyyy-MM-dd>.txt` applies only to this skill-development repository. It must not be copied into target-project behavior unless that target project defines the same rule.
38. Canonical report path is `.agrimap-agent/reports/YYYY-MM/<run_id>-<context>.md`. Reports are human-readable terminal summaries for light, standard, and regulated work and are generated from authoritative artifacts/logs.
39. Do not create `tasks/**/execution_log.md`. Do not create or link hidden-reasoning artifacts such as `agents/simulation/**/reasoning_*`. A user-visible simulation deliverable may contain evidence and conclusions but never hidden chain-of-thought.
40. Reports include execution ID, optional task ID, conversation/session identity, requester, identity source, model, role, agent, provider, workflow depth, start/finish/duration/status, linked recent memory/raw prompt/daily log/task when applicable, objectives, material files, issues, task-file status, summary, and completion checklist.

## Boundaries

41. Memory must not contain full transcripts, raw tool output, every prompt invocation, step-by-step agent activity, hidden chain-of-thought, or unsourced/stale facts.
42. Tasks must not contain unpromoted permanent knowledge, raw telemetry, copied external prompt templates, or a whole-project changelog.
43. Logs must not become project documentation, always-loaded memory, a prompt-template store, or long human-facing results.
44. Raw prompts, generated instructions, machine logs, resumable memory, task contracts, and human reports have separate owners and directories.

## Acceptance and migration

45. Existing legacy artifacts remain readable in place. New writes use this version's schema after the runtime change; no destructive bulk migration is required.
46. Focused and full tests cover artifactless light lifecycle, unchanged standard/regulated semantic gates with new paths, blocked/cancelled/completed transitions, memory promotion, daily-log concurrency/history, raw-prompt separation, reports, legacy compatibility, and generated-provider synchronization.
47. Final delivery includes a coverage matrix mapping every requirement in this version to implementation and verification evidence.
