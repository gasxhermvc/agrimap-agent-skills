## Requirements carried forward from Version 2

1. Every `depth=light` workflow persists durable evidence only under `.agrimap-agent/logs/**` and `.agrimap-agent/memory/**` and creates no per-task directory or brief/checklist/qa/result artifact.
2. Ignored `runtime/**` markers may remain operational state but are not durable evidence.
3. Light start/checkpoint/validate/complete/blocked/cancelled/history paths work from logs/memory; terminal evidence includes requester, executor, files, verification, outcome, reason, and concerns.
4. Standard and regulated workflow artifacts and validation remain unchanged.
5. Existing task history is never deleted or destructively migrated.
6. Runtime, schema, lifecycle/operation contracts, hooks, tests, docs, and generated mirrors remain synchronized.
7. Existing uncommitted SQL preflight files and `AGENTS.md` must not be overwritten by unrelated edits.
8. No commit, publish, release, global-cache mutation, or destructive deletion is authorized during product implementation.

## Golden and preflight requirements added in Version 3

9. Golden artifacts are teaching examples/templates and provenance evidence. Incoming user/project input is never required to match a golden file or its hash.
10. Normalized pattern contracts, owner decisions, active schema/callers/tests, and deployed compatibility govern transformation. Golden examples supply optional structural traits only after applicability is established.
11. `raw-immutable` golden material is analysis/provenance evidence and is not selected as a released-runtime template by default.
12. Raw-byte hash, manifest coverage, duplicate-content, and package-integrity verification belongs exclusively to development/CI/package validation.
13. Released runtime preflight must not import, call, or depend on `verifyGolden`, `HASH_MISMATCH`, or package-wide golden integrity state.
14. Runtime preflight validates only task input identity and selects an optional guidance pack. Missing/inapplicable examples produce actionable warnings or fallback to normalized rules; they do not globally block work.
15. Hard runtime blocks are limited to invalid required input, missing authoritative schema/caller evidence when necessary, or unresolved material behavior/data ambiguity.
16. Update conflict precedence, SQL guidance, operation contracts, tests, docs, and generated mirrors so no text says golden examples override an applicable active contract or that integrity failure blocks runtime inspection/generation.
17. Keep `npm run verify:golden` strict and passing as a dev/CI/package command.
18. Deliver a safe PowerShell cache-clear command that resolves the exact Codex plugin cache target under `%USERPROFILE%\.codex\plugins\cache\agrimap-agent-skills`, verifies it remains under the cache parent, and only then removes that target recursively. Document that it must be used after the fixed package is installed/reinstalled; do not execute it in this task.

## Changes from Version 2

- Added the owner-approved separation of golden examples from normative/runtime integrity, removal of `verifyGolden` from released preflight, runtime fallback rules, and a safe post-install cache-clear command.
