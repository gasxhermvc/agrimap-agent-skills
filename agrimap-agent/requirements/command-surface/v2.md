## Requirements carried forward from Version 1

1. `agm-fe` supports `analyze`, `design`, `create`, `edit`, and `test`.
2. `agm-be` supports `analyze`, `design`, `create`, `edit`, and `test`.
3. `agm-sql` supports `analyze`, `design`, `create`, `edit`, and `explain`.
4. `agm-refactor` routes `target=fe|be|sql` with explicit behavior mode.
5. Prefer the verb `analyze`; do not expose `analyzer`.
6. Domain façades resolve one action before inspection or product writes.
7. `analyze`, `design`, and SQL `explain` are product-read-only. `create`, `edit`, and explicitly authorized test/refactor work are product-write.
8. Passive activation never grants write authority. Direct bounded work remains proportional; broader tracked execution uses the regulated workflow.
9. Generated provider adapters, operation entrypoints, lifecycle/elicitation rules, documentation, validation, and tests remain synchronized.

## Version 2 command decisions

10. Keep `agm-analyze` callable.
11. Keep and promote `agm-design` as a unified design skill covering `target=fe|be|sql|architecture`; it routes to the applicable discipline without granting write authority.
12. Remove `agm-create-feature`, `agm-create-unit-test`, `agm-refactor-fe`, `agm-refactor-be`, and `agm-refactor-sql` from config registration, generated commands, plugin skill inventory, operation entrypoints, help, tests, documentation, and installed-package output.
13. Removal means the five skills no longer appear in host skill lists. Preserve only non-executable migration documentation/history mappings:
    - create/edit → `agm-fe|agm-be|agm-sql`
    - target refactor → `agm-refactor target=fe|be|sql`
    - unit tests → passive FE/BE/refactor test decision policy
14. Historical log/task operation names remain readable strings and do not require executable compatibility aliases.

## Passive unit-test policy

15. Unit-test assessment is a passive capability inside already authorized FE/BE/refactor implementation; it is not a standalone skill.
16. Record one decision: `required|recommended|not_applicable`, with evidence and reason.
17. Tests are `required` when touched scope changes behavior, data semantics, a public contract, a bug regression, multiple/failure branches, concurrency/security behavior, or a shared component with an existing test harness.
18. Tests are `recommended` for lower-risk testable work; the user decides whether to expand scope.
19. Tests are `not_applicable` only with a concrete reason.
20. Read-only analyze/review/diagnose/design never writes tests through passive activation.

## Acceptance

21. Primary generated help contains `agm-design` as the unified cross-discipline design route and retains `agm-analyze`.
22. The five removed aliases are absent from source/generated package inventories and stale generated outputs are cleaned during sync.
23. Tests prove unified design routing, passive test classification, no passive write escalation, migration documentation, and removal from all provider surfaces.
