# Requirement: Domain-oriented command surface — v1

## Context

The current public commands are organized around implementation activities. `agm-create-feature` also conflates creation and modification, while design, unit-test creation, and target-specific refactor aliases make the surface larger than the user's domain-oriented mental model.

## Public command surface

1. Add `agm-fe` with `analyze`, `design`, `create`, `edit`, and `test` actions.
2. Add `agm-be` with `analyze`, `design`, `create`, `edit`, and `test` actions.
3. Add `agm-sql` with `analyze`, `design`, `create`, `edit`, and `explain` actions.
4. Add one correctly spelled `agm-refactor` command with `target=fe|be|sql` and the existing explicit behavior modes.
5. Prefer the verb `analyze`; do not expose the noun `analyzer` as an action.

## Passive capabilities

1. Design becomes a read-only passive discipline used by `agm-fe`, `agm-be`, `agm-sql`, and `agm-create-prompt`.
2. FE/BE test advising passively evaluates behavior branches, failure paths, regressions, and the existing test framework, then proposes useful tests.
3. Passive test advising never writes product files. Test creation requires explicit `action=test` or an unambiguous user request to create tests.
4. SQL explain is read-only. It explains purpose, inputs/outputs, reads/writes, flow, transactions, errors, assumptions, risks, and performance using `FACT`, `INFERENCE`, and `UNKNOWN` evidence labels. It never executes SQL or edits files; modification routes to `edit`.

## Permission and routing boundaries

1. A domain façade must resolve exactly one action before product inspection or writes.
2. `analyze`, `design`, and `explain` are product-read-only.
3. `create`, `edit`, and explicit `test` are product-write operations.
4. Passive activation never grants product-write authority.
5. Direct domain writes remain bounded; work exceeding the direct lifecycle routes to `agm-create-prompt` and later `agm-exec`.

## Compatibility migration

1. Keep `agm-analyze`, `agm-design`, `agm-create-feature`, `agm-create-unit-test`, and target-specific `agm-refactor-fe|be|sql` as deprecated compatibility aliases for existing prompts and automation.
2. Remove deprecated aliases from the primary generated operation index/help surface.
3. Each compatibility contract identifies its replacement without silently changing the recorded historical operation name.

## Acceptance

1. Generated Codex/Claude/Gemini adapters expose the four new public commands.
2. Generated operation entrypoints show action, mode, activation, and allowed-depth boundaries.
3. Lifecycle and elicitation rules enforce action resolution and passive no-write behavior.
4. Documentation teaches the domain commands first and labels old commands deprecated.
5. Validation and automated tests cover public visibility, compatibility aliases, passive behavior, SQL explain, and unified refactor routing.
